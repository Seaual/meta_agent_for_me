# Strategic 视角分析

## 核心价值主张

将技术项目代码自动转化为结构化的交互式教程，让开发者快速上手新项目或新框架。

---

## 成功指标

1. **教程完整性**：生成的教程包含分步指南 + 代码示例 + 练习题，三要素齐全
2. **语言覆盖**：支持 Python 和 JavaScript 两大主流语言
3. **反馈闭环**：最多 2 轮审查确保内容质量

---

## 边界定义

**在 team 内**：
- 扫描项目结构，识别关键代码文件
- 生成分步指南和代码示例
- 设计配套练习题
- 审查并改进教程内容
- 组装最终交付物

**在 team 外**：
- 用户需提供项目源代码（不在 team 内生成）
- 最终输出为 Markdown 文档（不负责渲染或发布）
- 不涉及运行代码或验证代码正确性

---

## 最小可行架构

| Agent | 职责 | 必要性 |
|-------|------|--------|
| file-analyzer | 扫描项目结构 | 核心：输入源头 |
| content-writer | 编写教程正文 | 核心：主要产出 |
| exercise-designer | 设计练习题 | 核心：交互性保障 |
| content-reviewer | 审查内容质量 | 核心：质量保障 |
| assembler | 组装最终文档 | 核心：交付保障 |

5 个 agent 均为必要，无法进一步简化。

---

## 扩展路线

- **v2 可能加入**：
  - 支持更多语言（Rust、Go）
  - 多格式输出（HTML、PDF）
  - 与文档站点集成（MkDocs、Docusaurus）

- **v3 可能加入**：
  - 自动运行代码示例验证正确性
  - 根据用户反馈优化教程结构
  - 多语言翻译支持

---

## 战略风险

| 风险 | 缓解建议 |
|-----|---------|
| 项目结构复杂导致分析不全 | file-analyzer 优先识别入口文件和核心模块 |
| 练习题与教程内容脱节 | exercise-designer 依赖 file-analyzer 输出，确保上下文一致 |
| 审查循环过多影响效率 | 最多 2 轮限制，避免无限迭代 |
