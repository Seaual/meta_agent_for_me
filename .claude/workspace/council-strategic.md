# Council Strategic Analysis

## 核心价值

这个 Agent Team 的本质是一个**科研产出导向的学习加速器**，而非简单的"学习规划工具"。

用户的核心痛点不是"不知道学什么"，而是"如何在极短时间内完成跨领域转型并产出高质量科研成果"。时间紧迫性（上半年）和目标高要求（SCI 一区）构成了核心约束。

因此，Team 的价值主张是：

**帮助用户在 4-6 个月内，从零基础到具备独立完成 SCI 一区级别研究的能力，通过系统化的知识图谱、精准的资源筛选、里程碑式的时间管理，最大化学习 ROI。**

关键洞察：
- 用户有 LLM agent 背景，这是潜在的技术交叉优势
- SCI 一区要求高，Team 需要帮助识别"高价值研究切入点"
- 每日 6 小时学习时间是硬约束，学习效率比广度更重要

---

## 边界定义

| 应该做 | 不应该做 |
|--------|---------|
| 构建低空经济 + TSP + 路线规划的知识图谱 | 代写论文或代做实验 |
| 设计阶段性学习目标与里程碑 | 保证论文发表结果 |
| 筛选高质量文献和开源代码资源 | 代替用户实际学习 |
| 提供研究切入点建议（基于热点和可行性） | 提供超过用户能力的复杂实验方案 |
| 整合用户的 LLM agent 背景，评估交叉创新机会 | 强制要求用户融合原有技术 |
| 输出结构化的学习路线文档 | 提供过于细化的每日时间表 |
| 识别关键论文和必读文献 | 提供泛泛的资源清单 |
| 标注时间节点和检查点 | 承担论文投稿流程责任 |

### 边界判断依据

1. **科研诚信红线**：不代写、不作弊，Team 是"教练"而非"运动员"
2. **可控性原则**：Team 只能控制学习路线质量，不能控制论文发表结果
3. **用户自主性**：最终的研究方向选择、学习执行由用户决定

---

## 成功指标

| 指标 | 衡量标准 |
|------|---------|
| 知识图谱完整性 | 覆盖低空经济核心概念、TSP 问题分类、主流算法族 |
| 学习路线可行性 | 每个阶段目标在用户时间预算内可实现 |
| 资源筛选精准度 | 推荐文献/代码与用户研究目标高度相关 |
| 研究切入点价值 | 至少提供 2-3 个具备 SCI 一区潜力的选题方向 |
| 里程碑清晰度 | 每个里程碑有明确的完成标准和检查点 |

## 扩展路线

### v1（当前版本）—— 学习路线规划

- 领域知识图谱构建
- 学习路径设计
- 资源搜集与筛选
- 时间规划与里程碑设定

### v2（中期）—— 研究执行辅助

- 论文写作结构建议
- 实验设计框架指导
- 文献综述自动生成
- 投稿期刊推荐

### v3（长期）—— 全流程科研助手

- 论文草稿评审
- 实验代码模板生成
- 学术社区动态追踪
- 合作者推荐

## Agent 数量建议

**建议：4 个 Agent（精简配置）**

| Agent | 职责 | 理由 |
|-------|------|------|
| **domain-researcher** | 低空经济 + TSP + 路线规划知识图谱 | 核心领域知识，决定学习内容的准确性 |
| **learning-designer** | 学习路径设计 + 资源筛选 | 串联领域知识到可执行计划 |
| **resource-scout** | GitHub 代码 + 学术文献搜集 | 独立的资源获取职责，需网络权限 |
| **milestone-planner** | 时间规划 + 里程碑设定 + 检查点 | 保障时间约束下的执行力 |

**不建议第 5 个协调 Agent 的理由**：

1. 用户数量少（单人使用），交互简单
2. 4 个 Agent 的职责边界清晰，无需额外协调层
3. 精简配置降低复杂度，符合 minimal profile

---

## 战略风险

| 风险 | 影响 | 缓解建议 |
|------|------|---------|
| 时间过于紧迫，无法完成研究 | 用户目标落空 | 在里程碑中设置"最小可行研究目标"作为保底 |
| 跨领域学习曲线陡峭 | 学习效率低 | 建议优先掌握核心概念，建立最小知识闭环后再扩展 |
| SCI 一区要求被低估 | 选题过于简单 | 在研究切入点建议中明确标注"创新点"和"贡献度"评估 |
| LLM agent 背景未有效利用 | 浪费用户优势 | domain-researcher 主动识别交叉创新机会并标注 |

---

## 对需求预估值的支持

需求预估的 4-5 个 Agent 数量合理。本分析建议采用 4 个精简配置，与预估下限一致。如果后续发现资源搜集工作量过大，可考虑拆分为"代码搜集"和"文献搜集"两个 Agent，但当前不建议过度设计。
